Overview

One assistant for the whole migration journey

This project delivers an AI-powered Migration Assistant that helps teams assess, plan, and execute application migrations. It uses a two-role flow: Application Users submit migration requests (app profile data); Admins run research, generate and edit reports, and download. It combines a vector-backed Knowledge Base (KB) with optional LLM summarization and a phased roadmap from ideation to production.

The assistant supports migration practitioners and CoE teams with semantic search over migration artifacts, chat over the KB, and structured workflows for assessment and planning.

Design

How it works

Architecture & Design (umbrella) – Single entry point for all design docs: decision matrix, wireframes, assessment design, admin design, deployment.

Knowledge Base: The KB is built on a Vector DB (Pinecone) with integrated embeddings. User queries are run against the index; optionally, retrieved chunks are sent to an LLM for summarization into a single answer (RAG).

Assessment (Migration Request): Two roles: Application Users fill the profile and submit for assessment (no research or report); Admins view submissions, run research (KB + optional manual input), generate reports, edit/enhance, and download. Multi-agent system (Research Agent, Summarizer) runs in the Admin flow only.

Roadmap

Project timeline & story

A four-weekend journey from ideation to a production-ready Migration Assistant—each phase builds on the last with clear outcomes.

1
Weekend 1

Ideation, design & Knowledge Base

We started with a clear goal: give migration teams one place to search and chat over their documents. Initial ideation and design led to implementing the Knowledge Base—vector index, document ingestion (PDF, DOCX, XLSX, CSV, PPTX), semantic search, and basic chat with optional LLM summarization.

Outcome: Working KB with Pinecone, embeddings, and RAG-ready chat.
2
Weekend 2

Assessment multi-agent RAG

With the KB in place, assessment uses a two-role flow: Application Users submit migration requests (profile); Admins run multi-agent RAG (research over KB, summarization, report generation), edit reports, and download. This keeps the flow scalable and separates request intake from CoE execution.

Outcome: Two-role migration request flow with Admin-driven research and report generation.
3
Weekend 3

PoC to production

We hardened the system for production: guardrails, security, scalability, deployment, Dockerization, and observability (logging, metrics, alerts). The assistant moved from “it works on my machine” to “ready for teams.”

Outcome: Production-ready deployment and observability.
4
Weekend 4

Functionality & enhancements

We added polish and extensions: planning module improvements, migration implementation and support placeholders, UX refinements, and documentation. The result is a cohesive CoE assistant you can extend further.

Outcome: Full CoE assistant with docs and UX polish.

Lessons learnt

Challenges and how we fixed them

Issues we faced during development and the approaches that worked.

Start small: local first, then scale

We focused on local deployment and a working end-to-end flow before scaling to cloud deployments and packaging. This avoided big-bang complexity and made it easier to debug and iterate. Moving to Docker and production came only after the core KB, search, and chat were stable.

PDF and DOCX aren’t supported by default

Vector indexes and embedding APIs work on text; PDF and Word (.docx) are binary formats, so they can’t be indexed without extraction. We added a document extractor layer that uses appropriate libraries (e.g. PyMuPDF for PDF, python-docx for DOCX) to pull out plain text (and structure where useful) before chunking and embedding. That way all project documents can be ingested into the Knowledge Base.

Field mapping (text vs content) and re-indexing token issues

We hit mismatches between the field names used when indexing (e.g. text or content) and what the search/rerank pipeline expected, which led to empty or wrong results. We fixed it by standardizing on a single field (e.g. content) for the body text and ensuring the frontend and backend both read that field. Re-indexing then surfaced token limit errors with the reranker (e.g. bge-reranker has a max input length); we added a fallback that retries search without rerank when the reranker fails due to token limits, so long chunks don’t break the flow.
