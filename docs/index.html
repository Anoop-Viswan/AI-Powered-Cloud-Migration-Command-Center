Build a Cloud Migration Command Center

Single pane of glass to streamline all your cloud migration tasks—assessment, planning, and execution with a vector-backed Knowledge Base and optional LLM summarization.

Overview

One assistant for the whole migration journey

Cloud Migration Command Center (CMCC) is an AI-powered migration command center: a Knowledge Base (semantic search over migration docs via Pinecone), Assessment (app profile → research → report with optional Tavily for official-doc search), and Admin & Diagnostics (run research, view reports, monitor LLM/tool usage). Admin screens are protected by username/password; the app is deployable to cloud (e.g. Render) with CI/CD via GitHub Actions.

The system uses a two-role flow: Application Users submit migration requests (profile data, validation); Admins run research (KB first, then optional web search), generate and edit reports, run quality checks, and download. All design docs, wireframes, and this project narrative are in the documentation section below.

Modules

What’s inside

The system is organized into the following modules:

📚

Knowledge Base (KB)

Vector-backed semantic search over migration documents (PDF, DOCX, XLSX, CSV, PPTX, text). Pinecone index, embeddings, optional LLM summarization (RAG). Admin controls for indexing and configuration; spend guardrails.

🔍

Assessment (Migration Request)

Two-role flow: Application Users submit app profile (mandatory fields, validation, sanity checks); Admins run research (KB confidence + explainability, optional Tavily for official docs), generate reports, target-state diagram (LLM + HITL), quality check (score + reasons), edit, and download.

📊

Admin & Diagnostics

Admin command center: list of assessments, run research/generate report, KB config. Diagnostics tab: track LLM, Tavily, Pinecone calls; duration; approximate cost; configurable thresholds and alerts; request log and drill-down. Optional LangSmith tracing.

🔐

Admin authentication

Admin screens protected by username/password (env: ADMIN_USERNAME, ADMIN_PASSWORD). Session cookie; login/logout/me endpoints. TOTP/MFA planned for a later iteration.

📋

Planning & Execution

Placeholders for future runbooks, checklists, and post-migration artifacts.

Design

How it works

Knowledge Base: Documents are extracted, chunked, and embedded; the Vector DB (Pinecone) serves semantic search. Queries return chunks; optionally an LLM summarizes them into a single answer (RAG).

Assessment: Application Users complete the profile (overview, architecture, data, BC/DR, security); validation and sanity checks run before submit. Admins see submissions, run research (KB first with confidence and “why match,” then optional Tavily for official docs), then generate report (narrative + target-state diagram). Diagram uses an LLM design phase, optional human-in-the-loop clarification, then Mermaid → PNG; report includes quality check (score per criterion). Re-run research or regenerate report clears prior artifacts and rebuilds.

Roadmap

Project timeline & story

Iterative journey from ideation to a production-ready Migration Command Center—design first, then implementation; each phase added transparency, explainability, and operational readiness.

1
Phase 1

Foundation, flexibility & Knowledge Base

LLM provider abstraction (OpenAI, Anthropic, Azure); umbrella design doc with links to wireframes and decision matrix. Knowledge Base: Pinecone vector index, document extractors (PDF, DOCX, XLSX, CSV, PPTX), semantic search, chat with optional LLM summarization. Document extractors for binary formats; field standardization (e.g. content) and reranker token-limit fallback.

Outcome: Working KB and flexible LLM/config; design docs consolidated.
2
Phase 2

Assessment, two-role flow & research

Assessment module: mandatory fields, validation per section, RTO/RPO hints, sanity and LLM-based content validation. Two-role pivot: Application User (submit only) and Admin (view, research, report). Research flow: KB first with confidence and explainability (“why match”); below threshold → Tavily for official docs; streaming SSE; Tool Gateway for extensibility. Report generation with quality check (score + reason per criterion); target-state diagram (LLM design phase, HITL clarification, Mermaid → PNG). Cleanup on re-run research/regenerate report.

Outcome: Two-role flow, research with transparency, quality check, diagram, and full cleanup.
3
Phase 3

Config, feature status & Diagnostics

Feature-status API so the UI shows when LangChain/LangSmith/Tavily are unavailable (API keys, limits). .env load-once, reload-on-demand; exact Tavily errors surfaced. Diagnostics tab: track LLM, Tavily, Pinecone—call counts, duration, approximate cost; configurable thresholds and alerts; request log with drill-down; usage patterns. Optional LangSmith tracing.

Outcome: No silent failures; in-app diagnostics and observability.
4
Phase 4

Deployment, CI/CD & Admin auth

Docs reorganized (Architecture-and-Design, Setup-and-Reference, Deployment, Guides, Development-Iterations); rebrand to Cloud Migration Command Center. Docker image; deploy to Render with env-based config and secrets. GitHub Actions CI: backend tests, frontend build; optional interface-checks job (verify_setup.py) when repo variable set. Branch protection so merge requires green CI. Admin authentication: username/password, session cookie, login/logout/me; AdminGuard and login page in frontend.

Outcome: Cloud deployment, CI/CD, and protected Admin screens.

Lessons learnt

Challenges and how we fixed them

These lessons come from the iterative, AI-assisted development of this project. They reflect the kinds of guidance that shaped the product and what worked when things went wrong.

Design before code

When behavior or flow is non-trivial (research flow, diagram, tools), lock the design in writing first. Ask for a blueprint or design doc and approve it before implementation. We did this for the research flow, diagram design, and tool gateway; it avoided rework and kept behavior consistent.

Transparency over silence

All steps need to be informed—no silent failures. We added a feature-status API so the UI shows when LangChain/LangSmith/Tavily are not running (API keys, limits). Live SSE steps show what the LLM and tools are doing (timings, key results). Exact Tavily (and other) errors are surfaced instead of generic messages.

Explainability for decisions

Users and operators need to see why something happened. The quality check returns a score and a reason per criterion (comprehensive, actionable, useful, diagrams). KB hits include “why match” (origin/destination, features). Research shows key results before synthesis so you can see what was extracted and used.

Professional standards, not generic output

When the target-state diagram was “nonsense,” we switched to an LLM-first design phase with reference-architecture guidelines, then Mermaid → PNG. We deliver real artifacts (e.g. .mmd + PNG) and embed them in the report and DOCX, not just markdown.

Clean state on re-run

When the user runs research again or regenerates the report, clear everything and rebuild. We clear approach, report, QC, research_details, and diagram folder on “Run research”; clear report and QC on “Generate/Regenerate report.” Re-running never leaves stale data or UI.

Human-in-the-loop when accuracy matters

If the model might need more data (e.g. for the diagram), expose clarification questions in the UI and let the user answer or “proceed anyway.” We use this in the diagram design phase so reports stay accurate when the profile is ambiguous.

Validation and sanity checks

Mandatory fields, section-level validation, and LLM-based content checks prevent junk input. We added RTO/RPO hints, validation gates before submit, and sanity checks so unreasonable values (e.g. 10000 h RTO) are flagged and confirmed with the user.

Config and .env clarity

Load .env once at startup; provide a “reload .env” endpoint so config changes don’t require a full restart. Document where every variable comes from (local .env, Render env, GitHub Secrets/Variables for CI). We documented this in ENV-Reference and Deploy-Render so deployers know exactly what to set.

Extensibility from day one

A standard tool gateway (direct, MCP, UTC) makes it easier to add Tavily, Teams, WhatsApp, email later. We put Tavily and LLM behind the same patterns and wrote an extension guide so new tools follow a consistent contract.

CI: secrets vs variables

In GitHub Actions, use Secrets for API keys and Variables for flags (e.g. RUN_INTERFACE_CHECKS). Job-level if cannot depend on secrets in some contexts; use a repo variable to gate the interface-checks job. Pass optional config (e.g. OPENAI_MODEL) via variables so CI can run with defaults or overrides.

Tests and protected routes

When we added admin auth, tests that call admin/diagnostics endpoints started getting 401. We disable admin auth in tests by clearing ADMIN_USERNAME and ADMIN_PASSWORD in the test client fixtures so the same app instance serves both protected (production) and unprotected (test) behavior.

Document extraction and token limits

Vector indexes need text; PDF/DOCX are binary. We added a document extractor layer (PyMuPDF, python-docx, etc.) before chunking and embedding. We also standardized on one field (e.g. content) for body text and added a fallback when the reranker hits token limits so long chunks don’t break search.
