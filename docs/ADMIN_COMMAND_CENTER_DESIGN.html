1. Purpose

An Admin-only dashboard with two distinct views:

  1. Business / Assessments – Assessment statuses, errors, progress, quick actions
  2. System / Agent diagnostics – LLM invocations, tokens, data transfers, agent performance, request-level metrics

Audience: CoE admins, operators, support. Not end users running assessments.

2. Two-tab structure

The Admin page uses tabs (or sub-routes) to separate business tracking from system diagnostics:

TabRouteFocus
Assessments/admin or /admin/assessmentsBusiness: assessment list, status, errors, KB config
Diagnostics/admin/diagnosticsSystem: agents, LLM calls, tokens, data transfers, request-level
flowchart TB T1[Assessments tab] T2[Diagnostics tab] T1 --> A1[Assessments summary] T1 --> A2[KB health] T1 --> A3[Assessment list] T2 --> D1[LLM invocations] T2 --> D2[Tokens used] T2 --> D3[Request log]

5. Data sources for diagnostics

MetricSourceImplementation
LangSmithTraces, token usage, costs, latencyWhen LANGCHAIN_TRACING_V2=true; LangSmith UI. For our dashboard: local aggregation.
Local metrics storeOur own counters and logsPersist per-request: agent, tokens_in, tokens_out, latency_ms, status. SQLite table or JSON log.
OpenAI usageResponse headersusage.prompt_tokens, usage.completion_tokens; we log them.
PineconeRead unitsAlready in usage_tracker; extend for query count.
Recommendation: Add a local metrics store (e.g. metrics table) that we populate from each LLM/agent call. LangSmith remains the source of truth for deep traces; our dashboard shows aggregated metrics for quick visibility.

8. Visual design (command center)

  • Theme: Dark/slate (#0f172a, #1e293b); status colors (green/amber/red)
  • Tabs: Clear active state; Assessments | Diagnostics
  • Cards: Summary metrics in cards; tables for lists/logs
  • Refresh: Manual + optional auto-refresh (e.g. 30s for Assessments, 15s for Diagnostics)

9. Implementation order

  1. Tab structure – Admin page with Assessments | Diagnostics tabs
  2. Assessments tab – Backend GET /api/admin/assessments; panels (summary, list, KB, errors)
  3. Metrics instrumentation – Log per-request: agent, tokens, latency, status
  4. Diagnostics tab – Backend GET /api/admin/diagnostics/*; panels (LLM, tokens, agents, request log)
  5. Polish – Export, auto-refresh, retry/delete
