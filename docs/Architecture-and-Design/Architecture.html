1. Overview

The Cloud Migration Command Center is an AI-powered CoE (Center of Excellence) application that provides:

  • Semantic Search – Query the knowledge base with natural language
  • Migration Request (Assessment) – Two-role flow: Application users submit app data (profile); Admins run research, generate and edit reports, download. See Migration Request Flow.
  • Chat – Summarize and discuss documents
  • Admin – View submitted requests, run research, generate/edit reports, download; KB management; optional diagnostics

2. Research Flow (Assessment)

Admin-run research follows a defined flow: KB first with confidence and explainability; if confidence is low, official-documentation research (via Tool Gateway); all with live updates to the Architect. See Research Flow Blueprint for full detail.

Flow summary

  1. KB lookup – Structured queries (origin/destination, workload, use case). Search returns hits with scores and metadata (file_path, application).
  2. Confidence score – A single 0–1 (or %) “KB confidence.” Configurable low threshold (e.g. 0.35). Below threshold → inform Architect and trigger official-doc research.
  3. Explainability – For each relevant hit: source, score, why it matches (e.g. origin/destination same, features: stack, DB, RTO/RPO). Shown to the Architect.
  4. Low confidence path – If confidence < threshold: “Informing Architect” + research from official documents (e.g. Azure → Microsoft learn, Snowflake → Snowflake guides) via Tool Gateway (Tavily first, MCP optional).
  5. Findings and rationale – Official-doc results with URL, snippet, and rationale (“why this step should be followed”). Final approach document references both KB sources and official docs.
  6. Live updates – Streaming (e.g. SSE): “Searching KB…”, “KB done – confidence X%”, “Explaining matches…”, “Low confidence – searching official docs…”, “Summarizing…”, “Done.” UI shows a live progress panel (Cursor-style).

Tool integration

External tools (Tavily, future Teams/email) go through the Tool Gateway. Web search is implemented as a direct Tavily tool first; MCP support is optional. Same agent interface either way. See Tool Gateway Design and Tool Extension Guide.

4. LLM provider flexibility

The architecture supports seamless LLM switching via a provider abstraction. All LLM calls (Chat, Research Agent, Summarizer Agent) use a single get_llm() factory.

How it works

  • Provider selection: Set LLM_PROVIDER=openai|anthropic|azure_openai in .env
  • Unified interface: backend/services/llm_provider.py returns a LangChain BaseChatModel
  • No code changes: Switch providers by changing env vars and adding the relevant API key

Supported providers

ProviderEnv varsNotes
openai (default)OPENAI_API_KEY, OPENAI_MODEL, OPENAI_TEMPERATURE, OPENAI_MAX_TOKENSDefault
anthropicANTHROPIC_API_KEY, ANTHROPIC_MODELRequires langchain-anthropic
azure_openaiAZURE_OPENAI_ENDPOINT, AZURE_OPENAI_API_KEY, AZURE_OPENAI_DEPLOYMENTEnterprise / data residency

Adding a new provider

  1. Add a branch in backend/services/llm_provider.py → get_llm()
  2. Return a LangChain BaseChatModel
  3. Document env vars in .env.example
