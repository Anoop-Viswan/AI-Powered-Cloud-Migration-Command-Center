1. Summary table (with rationale)

Key decisions, candidates, final choice, and rationale for each.

Decision Candidates Final choice Rationale
Multi-agent orchestration LangGraph, LangChain LCEL, custom, AutoGen, CrewAI LangGraph Flow has conditional logic (Research fail → end; else → Summarize). LangGraph's graph model and conditional edges fit. LangChain LCEL is linear-only. Custom Python would reinvent state and tracing. AutoGen/CrewAI target conversational agents, not our sequential pipeline. LangGraph gives native LangSmith traces; used by LinkedIn, Uber, Klarna.
Observability LangSmith, custom, OpenTelemetry, Datadog LangSmith Built for LangChain/LangGraph. Zero config: set env vars, traces appear. Custom logging needs manual instrumentation. OpenTelemetry has experimental LangChain support. Datadog is general APM. LangSmith shows trace trees, token costs, latency per step—what Admin Diagnostics needs.
Vector DB / KB Pinecone, Weaviate, Milvus, Chroma, Qdrant Pinecone Project is pinecone-semantic-search; Pinecone already integrated. Switching would mean new index, embeddings, usage tracking. Weaviate/Milvus/Qdrant need self-host or cloud. Chroma is lighter but less scalable. No business reason to change.
LLM provider OpenAI, Anthropic, Azure, Vertex OpenAI (GPT-4o-mini) Chat/summarization already use OpenAI. Adding another provider increases config and testing. GPT-4o-mini balances quality and cost for synthesis. Anthropic has strong reasoning but different API. Azure/Vertex add enterprise setup. Consistency and simplicity favor OpenAI.
Web search Tavily, Serper, Exa, Bing Tavily We need clean snippets for LLM context, not raw HTML. Tavily is built for RAG/LLM use. Serper returns Google results but less curated. Exa is semantic search—different paradigm. KB-first strategy; Tavily only when KB sparse, so cost bounded.
State persistence SQLite, Postgres, JSON, in-memory SQLite MVP needs persistence without new infra. SQLite is single file, no server. Postgres overkill for current load. JSON has concurrency issues. In-memory loses data on restart. SQLite allows pluggable store; migrate to Postgres when scaling.
Async pattern Polling, SSE, WebSockets, Celery Polling Research/Summarize take 10–60s. Polling simplest: client calls GET every 2–3s. SSE needs streaming endpoints. WebSockets overkill. Celery needs Redis and workers. Minimal complexity for Phase 1; SSE later for better UX.
Report format Markdown, DOCX, PDF, HTML Markdown + DOCX LLMs output Markdown naturally. Preview MD in UI. For download, convert to DOCX via python-docx (already in use). PDF harder to generate. HTML less portable. MD + DOCX gives quick preview and standard business deliverable.
API pattern Step-by-step, run all Step-by-step Users need to see progress (Profile → Research → Report). Step-by-step lets them advance and retry individual steps. Single "run all" would be black box. LangGraph can run full flow internally; API stays step-by-step for clarity and debuggability.
Profile UI Single form, wizard, tabs Wizard + 7 pillar tabs Seven pillars would overwhelm a single form. Tabs group by domain. Wizard guides flow. "Continue to [next section]" enforces filling all pillars before Research. Matches real migration intake forms.
LLM provider flexibility Single provider, abstraction Provider abstraction get_llm() factory supports OpenAI, Anthropic, Azure. Switch via LLM_PROVIDER env var. No code changes. Supports enterprise (Azure data residency) and model choice (Anthropic).

2. Decision tree

Need orchestrator for multi-agents? → LangGraph (graph, conditional edges, LangSmith)
Need observability?                  → LangSmith (zero-config, trace trees)
Need vector KB?                      → Pinecone (project foundation)
Need LLM?                            → OpenAI (existing, LangChain support)
Need web search when KB sparse?      → Tavily (LLM-oriented snippets)
Need state persistence?              → SQLite (no infra, pluggable)
Need async for long ops?             → Polling (simplest; SSE later)
Need report output?                  → Markdown + DOCX (preview + download)

3. Multi-agent orchestration

Need: Coordinate Data Collector, Research, Summarizer with conditional logic and shared state.

Candidates: LangGraph, LangChain LCEL, custom Python, AutoGen, CrewAI

Analysis: Our flow has a branch: if Research fails, end; else continue to Summarize. LangChain LCEL is linear. Custom Python would require manual state and no tracing. AutoGen/CrewAI focus on conversational agents, not our sequential pipeline. LangGraph's StateGraph and conditional edges map directly to our flow.

Decision: LangGraph

Rationale: Explicit graph model matches flow. Conditional edges handle errors. Typed shared state. Native LangSmith. Industry adoption. Can run step-by-step or full graph.

4. Observability & tracing

Need: Trace agent calls, LLM invocations, token usage, latency for debugging and Admin Diagnostics.

Candidates: LangSmith, custom logging, OpenTelemetry, Datadog

Analysis: We use LangChain/LangGraph. LangSmith is built for this stack. Custom logging would need manual instrumentation. OpenTelemetry has experimental LangChain support. Datadog is general-purpose. LangSmith gives trace trees, token usage, cost per run with minimal setup.

Decision: LangSmith

Rationale: Zero-config with LANGCHAIN_TRACING_V2=true. Trace trees, token costs. Fallback: if not configured, tracing disabled. Aligns with Admin Diagnostics.

5. Vector database / Knowledge Base

Need: Semantic search over migration docs for Research Agent.

Candidates: Pinecone, Weaviate, Milvus, Chroma, Qdrant

Analysis: Project is pinecone-semantic-search. Pinecone already integrated. Switching would require new index, embedding pipeline, spend guardrails. Weaviate/Milvus/Qdrant need self-host or cloud. No strong reason to change.

Decision: Pinecone

Rationale: Project foundation. Existing search_knowledge_base(), namespaces. No new infra. Usage tracker and spend guardrails in place.

6. LLM provider

Need: LLM for Research synthesis and Summarizer report generation.

Candidates: OpenAI, Anthropic Claude, Azure OpenAI, Vertex

Analysis: Chat and summarization already use OpenAI. Adding another provider means extra config and testing. GPT-4o-mini sufficient for synthesis at lower cost. Anthropic would require separate integration. Consistency and simplicity favor OpenAI.

Decision: OpenAI (GPT-4o-mini)

Rationale: Already used. langchain-openai with LangSmith. Configurable via OPENAI_MODEL, OPENAI_TEMPERATURE, OPENAI_MAX_TOKENS.

7. Web search (when KB sparse)

Need: Fetch migration best practices when KB has little relevant content.

Candidates: Tavily, Serper, Exa, Bing

Analysis: We need clean snippets for LLM context, not raw HTML. Tavily built for RAG/LLM. Serper less curated. Exa is different paradigm. KB-first; Tavily only when sparse, so cost bounded.

Decision: Tavily

Rationale: Designed for LLM use. Simple API. KB-first; Tavily as fallback. Can swap if limits/cost become an issue.

8. State persistence

Need: Persist assessment state across requests.

Candidates: SQLite, PostgreSQL, JSON files, in-memory

Analysis: MVP needs persistence without new infra. SQLite is built-in, single file. Postgres overkill. JSON has concurrency issues. In-memory loses data. SQLite allows pluggable store; migrate to Postgres when scaling.

Decision: SQLite

Rationale: No extra infra. data/assessments.db. Pluggable store. Easy migration path.

9. Async pattern (long-running Research/Summarize)

Need: Research 10–60s, Summarize 5–30s; cannot block HTTP.

Candidates: Polling, SSE, WebSockets, Celery

Analysis: Polling simplest: client calls GET every 2–3s. SSE needs streaming. WebSockets overkill. Celery needs Redis. Phase 1: minimal complexity; SSE later for progress messages.

Decision: Polling

Rationale: Client polls until status completed or error. No streaming infra. Add SSE later.

10. Report output format

Need: Report preview in UI + downloadable file.

Candidates: Markdown, DOCX, PDF, HTML

Analysis: LLMs output Markdown naturally. Need both preview and download. DOCX is standard. python-docx already in requirements. Markdown for preview, DOCX for download covers both.

Decision: Markdown + DOCX

Rationale: LLM outputs MD. Preview in UI. Convert to DOCX for download. python-docx available.

11. API pattern

Need: Expose assessment flow to frontend.

Candidates: Step-by-step endpoints, single "run all"

Analysis: Users move through Profile → Research → Report. Step-by-step lets them see progress and retry individual steps. Single "run all" would hide intermediate state. LangGraph can run full flow internally; API stays step-by-step.

Decision: Step-by-step

Rationale: POST /start → PUT /profile → POST /research → POST /summarize. User advances explicitly. Easier retry and partial re-run.

12. Frontend assessment flow

Need: Guided UI for profile → research → report.

Candidates: Single form, wizard, separate routes

Analysis: Seven pillars would overwhelm a single form. Tabs group by domain. Wizard guides flow. "Continue to [next section]" ensures all pillars visited before Research. Aligns with real migration intake forms.

Decision: Wizard + 7 pillar tabs

Rationale: Route /assessment. Steps: 1. Profile (7 tabs), 2. Research, 3. Report. Section-to-section navigation; "Save & continue to research" only on final section.

13. LLM provider flexibility

Need: Ability to switch LLM providers (OpenAI, Anthropic, Azure) without code changes.

Approach: Provider abstraction via get_llm() factory in backend/services/llm_provider.py

Implementation: All LLM calls (Chat, Research Agent, Summarizer Agent) use get_llm(). Provider selected via LLM_PROVIDER=openai|anthropic|azure_openai. Each provider uses its LangChain integration. Adding a new provider = add branch in get_llm() + document env vars.

Decision: Provider abstraction

Rationale: Single abstraction point. No code changes to switch; only env vars. Supports enterprise requirements (Azure for data residency, Anthropic for alternative models). See ARCHITECTURE_DESIGN.html §3.
