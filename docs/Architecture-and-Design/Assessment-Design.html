Assessment Module

Architecture & Design Document

Multi-Agent System for Migration Assessments · Design Phase v1.0

1. Executive Summary

The Assessment module uses a two-role flow. Application Users fill the profile and submit for assessment (no research or report). Admins run research, generate and edit reports, and download. See Migration Request Flow.

Application User: Profile (7 pillars) → Validate & Submit → Confirmation. Admin: View submissions → Run Research → Generate Report → Edit/Enhance → Download.

6. API Design

REST endpoints. Application User uses start, profile, validate, submit. Admin uses list, get, research, summarize, report (edit + download).

MethodEndpointPurposeActor
POST/api/assessment/startCreate new assessment; returns assessment_idApp User
PUT/api/assessment/{id}/profileSave/update ApplicationProfileApp User
GET/api/assessment/{id}/validateValidate profile; returns { valid, errors, warnings, findings }App User
POST/api/assessment/{id}/submitSubmit for assessment (store, status=submitted, notify Admins). No research/report.App User
GET/api/admin/assessmentsList all assessments (filter by status); for Admin dashboardAdmin
GET/api/assessment/{id}Get full assessment state (profile, approach, report, status)Both
POST/api/assessment/{id}/researchTrigger Research Agent; returns ApproachDocumentAdmin
POST/api/assessment/{id}/summarizeTrigger Summarizer; returns reportAdmin
PUT/api/assessment/{id}/reportUpdate report (edit/enhance)Admin
GET/api/assessment/{id}/reportDownload report file (DOCX)Admin
Async pattern: Research and Summarize (Admin only) can take 10–60 seconds. Use polling: client calls GET /assessment/{id} every 2–3s until status advances.

6b. Profile validation flow

Application User flow: Profile must pass per-section and full validation. Validation runs when clicking "Continue to [next section]" and when clicking "Submit for assessment" (all pillars). On success, request is stored (status=submitted) and Admins notified; no Research or Report. Required: Overview (RTO, RPO, users, priority), Architecture (description or diagram), Data (DB migration Yes/No; if Yes then data volume, DB types, current DB description), BC&DR (all four), Security (auth, encryption). Cost and Project optional.

9. Implementation Phases

flowchart TB A1[Backend Router SQLite] --> A2[Research Agent] A2 --> A3[Summarizer Agent] A4[Frontend Wizard] --> A5[End-to-end MVP] A3 --> A5 A5 --> B1[Add Tavily] B1 --> B2[Tune queries] B2 --> C1[DOCX generation] C1 --> C2[Download endpoint] C2 --> C3[Polish]

Figure 8: Implementation phases

Phase 1 – MVP

  • Backend: Assessment router, SQLite store, ApplicationProfile schema
  • Research Agent: KB only (no Tavily)
  • Summarizer: Markdown report
  • Frontend: Assessment route, ProfileForm, ResearchStep, ReportStep, polling

Phase 2 – Web Research

  • Add Tavily to Research Agent
  • Logic: KB first; if sparse → Tavily

Phase 3 – Polish

  • DOCX report generation
  • Download endpoint
  • Optional: Clarification loop, SSE

10. Admin Command Center

Admin-only dashboard with two tabs: Assessments (business) | Diagnostics (system/agent/request level).

flowchart TB T1[Assessments tab] T2[Diagnostics tab] T1 --> A1[Assessments summary] T1 --> A2[KB health] T1 --> A3[Assessment list] T2 --> D1[LLM invocations] T2 --> D2[Tokens used] T2 --> D3[Request log]

Figure 9: Admin Command Center – two tabs

Tab 1: Assessments

Business tracking: assessment list, status, errors, KB config, seed. Panels: Assessments summary, KB health, Errors (24h), Assessment list, KB config & seed, Recent errors.

Tab 2: Diagnostics

System/agent/request level: LLM invocations, tokens used, agent calls, data transfers, latency (p95), request log. See docs/ADMIN_COMMAND_CENTER_DESIGN.md for full spec.

Visual design

Dark/slate theme (#0f172a), status colors (green/amber/red), monospace for IDs, card-based layout.

Access

URL-based: /admin is admin-only. No auth in Phase 1.
